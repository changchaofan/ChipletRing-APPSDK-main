# ChipletRing Android SDK Demo PRD

## 1. 文档说明

本文档基于 `Android/example/ringDemo`、`Android/source` 以及 `Android/library` 中的实际代码结构整理，定位为当前项目的产品需求文档（PRD）初稿。  
需要特别说明：

- 当前工程本质上是 `ChipletRing` Android SDK 的 Demo 与参考实现，不是面向普通消费者的完整商业 App。
- 文档中的能力描述以源码中已经出现的页面、权限、交互和 SDK 调用为准。
- `TestActivity`、`TestActivity2`、`TestActivity3` 中存在大量“测试按钮式”入口，代表 SDK 已具备或正在验证的能力，不等同于最终量产版 UI。

## 2. 产品概述

### 2.1 产品名称

ChipletRing Android SDK Demo

### 2.2 产品定位

这是一个用于对接智能戒指硬件的 Android 端演示项目，主要承担以下角色：

- 演示 ChipletRing SDK 的初始化、蓝牙扫描、连接、重连与设备控制能力
- 为合作方 App 或硬件团队提供集成参考
- 验证戒指在健康监测、原始数据采集、文件系统、HID 手势、NFC、音频、OTA 等能力上的可用性
- 作为 SDK 功能回归和联调测试载体

### 2.3 目标用户

- Android 客户端开发者
- 智能硬件/固件工程师
- 测试工程师
- 算法与数据团队
- 需要基于戒指能力做二次开发的合作方

### 2.4 使用平台

- Android
- `minSdkVersion 27`
- `targetSdkVersion 35`
- Java 17

## 3. 产品目标

### 3.1 核心目标

1. 提供一个可运行的 Android Demo，快速验证戒指连接与指令链路是否正常。
2. 对外展示 SDK 的主要硬件能力，降低合作方接入成本。
3. 提供历史数据、文件采集、睡眠分析、心率/血氧/温度/ECG 等能力的调用样例。
4. 提供 OTA、HID、NFC、实时音频/PPG 等高级能力的调试入口。

### 3.2 非目标

- 不追求面向 C 端用户的完整产品体验
- 不包含完整账号体系、云端后台、支付或商业化闭环
- 不保证当前 UI、文案和流程达到正式发布标准

## 5. 用户场景

### 5.1 开发接入场景

开发者首次接入 SDK，希望快速确认：

- App 是否能扫描到戒指
- 是否能稳定连接与重连
- 是否能正确读取设备版本、电量、时间、步数等基础信息

### 5.2 健康能力验证场景

测试或算法人员希望验证：

- 心率、血氧、温度、血压波形、ECG 等测量接口是否可用
- 历史数据是否能读取、解析和展示
- 睡眠算法与 GoMore 睡眠结果是否可对比

### 5.3 原始数据采集场景

数据团队希望：

- 在戒指端启动持续采集
- 按设定时长切片存储文件
- 从戒指批量下载数据到手机本地
- 进一步转存为 TXT/CSV 供算法分析

### 5.4 高级能力联调场景

合作方需要验证：

- OTA 升级是否可行
- HID 模式与手势是否正确上报
- 音频流、实时 PPG、6 轴传感器是否可被 App 接收
- NFC 能力是否能被工程集成

## 6. 核心流程

### 6.1 首次使用流程

1. 用户打开 App
2. App 申请蓝牙、定位、存储、通知等权限
3. App 扫描附近 BLE 设备
4. 通过广播数据识别 ChipletRing 设备
5. 用户点击目标设备进入测试页
6. App 发起连接并注册 SDK 回调

### 6.2 基础调试流程

1. 连接成功后进入 `TestActivity`
2. 用户可读取步数、电量、版本、时间等基础信息
3. 用户可触发心率、血氧、历史记录、系统控制等能力验证

### 6.3 采集与文件下载流程

1. 用户进入文件系统页面 `RingFileListActivity`
2. 可先格式化戒指文件系统
3. 配置总采集时长与单文件切片时长
4. 启动采集
5. 停止采集后获取文件列表
6. 下载单个/多个/全部文件
7. 本地保存并转为可分析内容

### 6.4 睡眠分析流程

1. 用户读取历史数据或 GoMore 睡眠数据
2. App 按时间序列整理睡眠分期
3. 页面以图表方式展示入睡、唤醒、深睡、浅睡、REM 等信息

## 7. 功能需求

### 7.1 设备扫描与发现

需求说明：

- App 启动后自动申请权限并发起 BLE 扫描
- 扫描时应过滤无效设备
- 通过 SDK 解析广播包，识别可接入的戒指设备
- 列表页支持下拉刷新重新扫描

当前实现依据：

- `MainActivity`
- `LogicalApi.getBleDeviceInfoWhenBleScan`
- `BLEUtils.startLeScan / stopLeScan`

### 7.2 设备连接与重连

需求说明：

- 用户点击设备后发起连接
- 支持已绑定设备直连
- 连接失败后自动延迟重试
- 页面生命周期内自动注册/释放 SDK 指令监听

当前实现依据：

- `TestActivity`
- `BaseActivity`
- `App.needAutoConnect`

### 7.3 基础设备信息

需支持：

- 获取步数
- 获取电量
- 获取版本信息
- 读取/同步时间
- 清空步数
- 修改蓝牙名
- 读取 RSSI

当前实现入口：

- `TestActivity`

### 7.4 健康测量能力

需支持：

- 心率测量
- 血氧测量
- 温度测量
- 血压波形读取
- ECG 数据采集与展示
- 实时 PPG 采集

当前实现入口：

- `TestActivity`
- `TestActivity2`
- `source/ecg/ElectrocardiogramActivity.java`

说明：

- ECG 页面通过 WebView 加载本地 HTML/JS 图表资源进行波形渲染。
- 实时测量结果多数以文本日志形式展示，适合调试，不适合终端用户产品直接复用。

### 7.5 历史数据读取与解析

需支持：

- 读取历史记录
- 展示读取进度
- 输出单条历史记录明细
- 清空历史数据
- 删除本地历史库
- 获取温度历史
- 基于历史数据生成 timeline

当前实现入口：

- `TestActivity`
- `TestActivity2`
- `HistoryListTempActivity`

### 7.6 睡眠分析

需支持：

- 通过本地算法计算睡眠结果
- 从服务端获取睡眠数据
- 集成 GoMore 睡眠总览与分期
- 用图表展示睡眠分段

当前实现入口：

- `TestActivity2`
- `GoMoreSleepActivity`

说明：

- 当前项目同时存在“本地睡眠计算”和“GoMore 睡眠数据”两条路线，说明项目处于多方案并行验证阶段。

### 7.7 文件系统与原始数据采集

需支持：

- 配置采集总时长
- 配置单文件切片时长
- 启动/停止采集
- 获取戒指文件列表
- 下载选中文件
- 一键下载全部文件并合并
- 格式化戒指文件系统
- 下载后保存为 TXT/CSV
- 提示用户打开本地导出目录

当前实现入口：

- `RingFileListActivity`
- `NotificationHandler`
- `LmAPILite`

业务价值：

- 这是当前工程里最接近“数据采集平台”产品形态的模块。
- 适用于离线采集、算法验证、实验数据留存。

### 7.8 采集参数配置

需支持：

- 定时启动采集
- 设置 PPG 频率
- 设置传感器频率
- 控制陀螺仪/加速度/温度/PPG 开关
- 设置 PPG RAW 数据采集时长
- 控制 PPG 自动采集

当前实现入口：

- `CollectionActivity`

### 7.9 HID 与手势能力

需支持：

- 设置 HID 模式
- 获取当前 HID 配置
- 获取 HID 能力码
- 支持手势/触摸功能查询与配置
- 支持配对、解配对、直连等 HID 相关流程

当前实现入口：

- `TestActivity2`
- `TestActivity3`

### 7.10 音频能力

需支持：

- 开启/关闭音频传输
- 支持 PCM / ADPCM 类型读取与配置
- 将音频字节流转为 PCM 并落地文件

当前实现入口：

- `TestActivity`
- `TestActivity2`

### 7.11 OTA 升级

需支持：

- 指定升级文件
- 检查版本是否需要升级
- 执行 OTA
- 展示升级进度、完成、失败、已是最新版本

当前实现入口：

- `TestActivity`
- `TestActivity2`

### 7.12 NFC 能力

需支持：

- NFC 页面入口
- 支持 `TECH_DISCOVERED` / `TAG_DISCOVERED` / `NDEF_DISCOVERED`
- 支持 Host Card Emulation 服务

当前实现依据：

- `AndroidManifest.xml`
- `NfcActivity`
- `MyHostApduService`

## 8. 页面与信息架构

### 8.1 页面结构

- `MainActivity`：设备搜索列表页
- `TestActivity`：基础功能测试页
- `TestActivity2`：扩展能力测试页
- `TestActivity3`：HID/配对测试页
- `RingFileListActivity`：文件系统与离线采集页
- `CollectionActivity`：采集参数配置页
- `GoMoreSleepActivity`：GoMore 睡眠展示页
- `HistoryListTempActivity`：历史温度记录页
- `NfcActivity`：NFC 调试页
- `ElectrocardiogramActivity`：心电图展示页

### 8.2 当前信息架构问题

- 页面入口较分散，依赖测试按钮串联
- 功能存在明显“研发调试态”特征
- 缺少统一导航和按业务分类的首页
- 部分文案乱码或编码异常，影响可读性

## 9. 非功能需求

### 9.1 性能

- BLE 扫描应在 5 秒内返回可发现设备
- 实时测量与波形展示应保持持续刷新
- 文件下载需支持大批量数据处理，避免 UI 卡死

### 9.2 稳定性

- 连接失败后支持自动重连
- 下载与解析过程应有异常保护
- 页面切换时应正确注册/反注册监听，避免回调泄漏

### 9.3 兼容性

- 兼容 Android 8.1 及以上
- 适配 Android 12+ 蓝牙权限
- 兼容 Android 13/14 媒体与外部存储权限变化

### 9.4 安全与权限

- 需要蓝牙、定位、通知、存储、NFC、网络等权限
- 涉及 `MANAGE_EXTERNAL_STORAGE`，正式商业版需谨慎评估合规性
- 项目中存在签名信息、Token、邮箱等测试配置，正式版本应迁移到安全配置体系

## 10. 关键依赖

- `ChipletRing` AAR SDK
- `st25sdk` 与 NFC 相关模块
- `XXPermissions`
- AndroidX 组件
- WebView + 本地 HTML/JS 图表资源

## 11. 当前版本已知问题与风险

### 11.1 产品层面

- 更像“SDK 能力集合 Demo”，而非结构清晰的正式 App
- 多数页面缺少业务化包装和状态管理
- 缺少统一异常提示、空态、加载态设计

### 11.2 工程层面

- 顶层缺少 README 和接入文档
- 存在中文乱码/编码异常
- 工程中保留较多测试常量、硬编码路径、固定版本号和示例 Token
- 顶层目录不是 Git 仓库根目录，版本协作信息不完整

### 11.3 商业化落地风险

- 如果直接面向终端用户发布，当前 UI/交互与权限策略仍不成熟
- 文件系统、原始数据、HID、OTA、NFC 等高级能力需要按具体业务重新裁剪
- 医疗/健康相关能力如果对外宣称，需要评估合规与准确性要求

## 12. 后续建议

### 12.1 如果目标是“SDK 对接 Demo”

- 保留现有测试页思路
- 增加 README、接入流程、能力矩阵、权限说明
- 按“连接/基础能力/健康能力/文件采集/高级能力”重组页面入口

### 12.2 如果目标是“合作方交付样板 App”

- 将测试按钮页改造成任务式页面
- 抽离设备连接状态管理
- 增加统一日志面板和错误码说明
- 清理硬编码与测试配置

### 12.3 如果目标是“终端用户产品”

- 重新设计信息架构和交互路径
- 只保留用户能理解的核心能力
- 将原始调试功能隐藏到开发者模式
- 建立账号、云同步、数据报告、隐私协议与合规体系

## 13. 结论

当前项目已经具备一个较完整的智能戒指 Android SDK Demo 雏形，覆盖从设备连接、基础信息读取，到健康测量、历史数据、原始文件采集、GoMore 睡眠、ECG、HID、音频、OTA、NFC 等多类能力。  
它的优势在于“能力面广、联调入口齐全”，短板在于“产品化不足、文档缺失、测试属性明显”。  
如果后续目标是对外提供 SDK 或合作样板工程，建议优先补全文档、梳理页面结构、清理测试配置，并将当前散落在多个测试页的功能重新按业务域整合。
